# STUDENT-HUB-05 - Contract Panel State Consistency

## Goal

This ticket touches only the Contract panel/state behavior.

## Blueprint Source

Use:

- `docs/blueprint/student-hub-state-edit-pattern.md`
- `docs/tickets/WORKFLOW-02-student-hub-state-blueprint.md`

- Student Information
- Intake and Program/Batch

- generation history if already implemented

## Generate Behavior

When admin/super_admin generates a contract:

- DOCX generation still works
- generated contract record/history still works
- application status updates if current logic already does this
- panel refreshes after generation
- latest generated contract appears without manual browser refresh
- generated history appears without manual browser refresh
- button/loading state is clear

## Role Rules

Admin/super_admin:

- can generate if contract is ready and current rules allow
- can see latest generated contract/history
- can download latest if available

Sales:

- cannot generate contract
- can view contract status if current app rules allow
- can view generated contract info only if current app rules allow

Viewer:

- read-only
- cannot generate contract
- download visibility should follow current app role rules
- do not expand viewer privileges in this ticket

## Refresh Pattern

Use the blueprint recommendation:

- API/server action updates database
- revalidate correct student hub path
- client calls `router.refresh()` after successful generation/action
- explicitly update local state if needed for latest generated contract display
- do not use `window.location.reload`
- do not rely only on browser cache/cookies

## Known Issue To Avoid

The panel should not keep showing stale readiness or old contract history after generation.

For example:

- if generation succeeds, panel should not still show old “ready” status only
- if latest generated contract exists, it should appear after generation
- if missing items are resolved, readiness display should refresh

## Implementation Notes

### What changed

Only one file was modified: `src/features/contracts/generate-word-button.tsx`

Changes:
- Added `useRouter` from `next/navigation`
- After successful contract generation and download, call `router.refresh()` to trigger RSC refetch
- Replaced `alert()` error display with inline error message state
- Added inline success message after generation
- Wrapped button in a div to support success/error message display below the button

### How it works

The API route (`src/app/api/contracts/generate-docx/route.ts`) already calls `revalidatePath` for the student hub path after generation. The missing piece was the client-side `router.refresh()` call. Now, after the fetch completes and the DOCX is downloaded:

1. `router.refresh()` triggers a RSC payload refetch
2. The server component re-runs `getStudentById`, which fetches fresh `contractGenerations`, `contracts`, and `applications` data
3. The contract panel re-renders with updated status, latest generated contract, and generation history
4. The application status badge at the top of the page also updates (e.g., from "Ready for Contract" to "Contract Generated")

### Role behavior

No role logic was changed. Existing rules:
- admin/super_admin: can generate contract when status allows (generate button visible)
- sales: cannot generate (generate button not rendered)
- viewer: cannot generate (generate button not rendered)

### Word template

`src/templates/contracts/student-enrolment-template.docx` was not modified.
